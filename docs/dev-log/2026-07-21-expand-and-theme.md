# ui-expand-context + Claude Code 配色（使用者追加需求）

2026-07-21 · commit `bcb85b3`

## 展開功能

每個變更點上下可拉入未修改的程式碼，讀起來像整份檔案，但變更點的分塊與打勾
完全不變。

**核心界線**：展開的行是脈絡，不是變更點——不可勾選、不可留 comment、不計入
任何進度、不進入 `j`/`k`/`u` 的巡覽順序。Reviewer 實測展開約 1830 行後，
標題與樹的計數與展開前**位元組相同**，`j`/`k` 走的仍是同一組 5 個變更點。

相鄰變更點之間 ≤8 行的小空隙直接顯示、不藏在按鈕後面（三行的空隙藏在點擊後面
比直接顯示更糟）。

檔尾的處理值得記：`/api/lines` 對超出範圍的請求是夾住而非報錯，所以問「最後一行
之後的 2000 行」會拿到 200 加上被夾到最後一行的內容。客戶端用 `n > belowLoaded`
過濾掉已顯示的那行，按鈕才會正確消失——否則會重複附加最後一行，且按鈕永遠可點。

## 配色

預設改為中性黑灰（`#191919`，R=G=B 完全相等，對比舊的 `#0f172a` 藍值高出 27）
+ 陶土色 accent `#d97757`。八個既有主題保留可選。

**順帶拆開一個既有的語意混淆**：原本綠色同時代表「已讀」「進度」「新增行」。
現在已讀與進度走 accent，diff 的新增/刪除維持綠/紅——後者是跨工具通用約定，
換掉會讓人讀錯 diff。使用者現在能一眼分辨「這行是加的」與「這段我看過了」。

## 一個靜默的 CSS 事故

實作中途 `unified` 按鈕整個消失。根因是實作者自己寫的 CSS 註解裡有一個字面上的
`*/`，提前關閉了註解區塊，**靜默作廢了整個 `--color-accent`**。

這類錯誤不會有任何警告，只會讓一堆看似無關的東西同時壞掉。修正後 reviewer
掃過整份樣式表：`/*` 與 `*/` 各 196 個完全配對，38 個 `var(--x)` 使用皆有對應
宣告，且在執行期逐一確認 38 個 token 都解析到非空值——不是抽樣。

## 對比度：量了，但量漏了一個維度

實作者依 brief 要求量測註解對三種底色（未變 / 新增 / 刪除）的對比，
全部通過 4.5：6.35 / 4.83 / 5.77。

Reviewer 發現量測本身有盲點：**已讀的變更點整塊淡化到 55% 不透明度**，
而「已讀」正是變更點在實際使用中最常處於的狀態。把淡化算進去後：

| 狀態 | 註解 vs 新增 | vs 未變 | vs 刪除 |
|---|---|---|---|
| 未讀（1.0） | 4.83 ✅ | 6.35 ✅ | 5.77 ✅ |
| 當前+已讀（0.75） | 3.41 ❌ | — | — |
| 已讀（0.55） | **2.47** ❌ | 2.82 ❌ | 2.69 ❌ |

不是這次造成的退步（舊主題在同樣狀態下是 1.82–2.00，更糟），但 brief 明白
要求「這次要量，不要看，不得重蹈覆轍」，結果量的**方式**漏了一個維度。

這是本專案反覆出現的同一種錯誤的變體：**驗證做了，但驗證的範圍不等於真實的
使用情境**。前四次是「測試在未修正的程式碼上也會通過」，這次是「對比度量了
但沒量最常見的狀態」。已另派修正。

## Review 的其他確認

- 用的是 diff 的 **target** ref（變更點行號所指的版本）。用 base 會靜默顯示
  另一個版本的程式碼，比什麼都不顯示更糟
- 展開的行注入 `<img onerror=...>` 等 payload 後渲染為惰性純文字
- 展開狀態不會在 tree 重繪後保留——但 `renderTree()` 只在 base/target 改變時
  觸發，那本來就是不同的 diff，重設是正確的。勾選框、折疊、切換檢視模式皆不觸發
- 連點三次只發出一個請求
- gutter 對齊：展開行與原生 diff 行的 gutter 寬度實測皆為 42px
